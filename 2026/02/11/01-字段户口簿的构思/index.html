字段户口簿的构思：给AI配上'业务词典'

DevCraft
2026-02-11
AI, AI业务词典, 数字化转型, 语义, 随笔

如果字段户口簿是「字典」，那需求文档生成的 Schema 就是「文章」。📖

AI 代码生成最大的瓶颈逐渐将不是模型，而是语义断层。

一、为什么需要字段户口簿？

1.1 同一个字段，三个系统的故事

在金融科技领域，尤其是券商系统中，字段混乱是 AI 落地的最大隐性成本。模型学的是公共语料，不知道我们内部叫 order_id 还是 trans_no。所以每次生成完都要人工校对字段名，等于 AI 只干了 30%的活。

产品经理写的”交易流水号”，在不同系统里有完全不同的名字：

系统 字段名
交易系统 serial_number
账务系统 ref_id
会计系统 voucher_no
风控系统 biz_trans_no

每次系统对接，都是一场”你翻译给我，我翻译给你”的人工消耗。20%-30%的人力耗在字段映射上。

1.2 四大痛点

痛点一：跨系统联调难

一个简单的订单详情页对接，需要：

  • 交易系统提供 order_id, status, amount
  • 账务系统返回 trans_no, order_state, trans_amount
  • CRM 系统同步 serial_number, state, total_fee

每次联调，开发人员需要逐个确认字段对应关系，测试人员需要验证数据准确性。2 天的联调工作，其中 1.5 天在做字段翻译。

痛点二：AI 代码生成不准

相信使用当前主流coding大模型后，你会发现：让当前最强的 AI 模型生成一个 CRM 相关列表页面。

结果：生成 10 次代码，7-8 次字段名是错的。

AI 模型再强，也无法知道：

  • 你们公司的”订单号”在交易系统里叫 order_id
  • 在账务系统里叫 trans_no
  • 在客服系统里叫 serial_number

没有语义字典，AI 只能靠”猜”。

痛点三：合规风险

金融行业对敏感数据有严格的监管要求：

  • 客户手机号、资金账号、证件号码都是敏感字段
  • 但在不同系统里，这些字段的标识方式完全不同
  • 有的系统用 phone，有的用 mobile，有的用 contact_no
  • 脱敏规则难以统一执行，合规审计难以追溯

痛点四：业务语义断层

这是最隐蔽也最致命的问题：

角色 使用的字段名
产品经理（PRD） 交易流水号
设计师（标注） 流水号字段
开发（实现） trans_no
AI 生成代码 order_id
测试环境 serial_number

四个不同的人，说的是同一件事，但用四个不同的名字。这就是业务与技术之间的”巴别塔”。

1.3 问题的本质

这不是技术问题，是语义不统一。

AI 模型学的是公共语料，它知道”订单”是什么意思，但它不知道你们公司内部把”订单”叫做什么。

二、字段户口簿是什么？

2.1 核心定位：字典 vs 文章

字段户口簿是「字典」，Schema 是「文章」。📖

这是理解整个项目的核心比喻：

字典的价值：

  • 唯一性：每个词条（字段）只有一个标准 ID
  • 权威性：告诉 AI”订单号”的标准写法
  • 兼容性：生成时就已经确定目标系统的字段名
  • 可追溯：记录谁、什么时候、为什么定了这个标准

Schema 生成时已完成解析：

  • hiagent 调用字段户口簿 API，将语义解析为标准字段 ID
  • Schema 里已经是正确的字段 ID，无需 mapping
  • 户口簿告诉 Schema：它的枚举值有哪些（让 AI 生成正确的下拉选项）
  • 户口簿告诉 Schema：它是否涉敏（生成代码时自动加脱敏）

2.2 不是什么

必须明确区分：

数据治理平台 字段户口簿
核心定位 企业级元数据管理 AI 代码生成的语义底座
用户 数据治理人员、分析师 PRD2Code、AI IDE、开发者
接口风格 面向人（UI、报表） 面向机器（API、毫秒级响应）
更新频率 低频（按需变更） 高频（随时同步别名）
产出 报表、文档、血缘图 API 服务、字段映射、枚举值

一句话总结：

字段户口簿不是数据治理平台的替代品，而是它的「AI 友好接口层」。

数据治理是给人看的报表和仪表盘，字段户口簿是给 AI 吃的数据服务。

2.3 核心价值

价值维度 具体体现
提效 联调时间从 2 天 →2 小时；AI 代码生成准确率从 70%→99%
降本 消除因字段混乱导致的返工，每月节省 20%-30%人力
合规 敏感字段统一标记，脱敏规则自动执行
赋能 AI 为 AI 提供”业务语义词典”，让 AI 真正理解公司数据
沉淀资产 字段定义、枚举值、映射关系成为公司级数字资产

2.4 核心定位金句

字段户口簿是「字典」，Schema 是「文章」 —— 最核心的比喻
给 AI 配的”业务词典” —— 最直白
AI 代码生成的语义底座 —— 技术圈
公司数据的”巴别塔翻译器” —— 有故事感
面向机器的数据中台 —— 战略高度

四、数据模型设计

4.1 核心实体

字段定义表 (field_definitions)

字段名 类型 说明
field_id string 标准字段 ID，如 order_id
display_name string 显示名，如”订单号”
domain string 业务域，如 order、customer
data_type string 数据类型：string/enum/decimal/datetime
sensitive_level int 敏感等级：0-3，0 为非敏感
mask_rule string 脱敏规则，如 middle4、prefix3
owner string 责任人/团队
status string 状态：draft/active/deprecated

系统别名表 (system_aliases)

字段名 类型 说明
field_id string 标准字段 ID
system_name string 系统名称，如 trade、accounting
alias string 该系统下的字段名
since date 生效日期
deprecated_at date 废弃日期（可选）

枚举值表 (enum_values)

字段名 类型 说明
enum_id string 枚举集 ID，如 ORDER_STATUS_V1
code string 枚举代码，如 2
name string 枚举名称，如”成功”
aliases json 各系统的别名映射
description string 描述

5.2 核心接口构思

5.2.1 MCP 工具（推荐 ⭐）

字段户口簿通过 MCP（Model Context Protocol）协议提供工具调用能力，AI Agent 可直接调用以下工具：

工具名 功能 使用场景
search_fields 关键词搜索字段 不确定用哪个字段时
resolve_alias 别名解析 产品经理写的”订单号”→order_id
get_field_detail 获取字段详情 需要类型、敏感标记、枚举时
get_enum_values 获取枚举值 生成下拉选项时
list_domains 列出所有业务域 需要了解有哪些域时

5.2.2 REST API（备选）

对于非 MCP 场景，也可通过 HTTP API 调用：

接口 功能 示例
GET /v1/field/{fieldId} 获取标准字段定义 /v1/field/order_id
GET /v1/resolve 根据别名解析标准字段 /v1/resolve?alias=trans_no&system=accounting
GET /v1/enum/{enumId} 获取枚举值定义 /v1/enum/ORDER_STATUS_V1?value=成功
GET /v1/sensitive/{fieldId} 获取字段敏感信息 /v1/sensitive/customer_phone
POST /v1/batch/resolve 批量解析 用于 AI 生成时的批量查询

5.4 MCP 方案设计

5.4.1 为什么选择 MCP？

对比维度 HTTP API MCP 工具调用
集成方式 轮询/定时 事件驱动
实时性 秒级 毫秒级
上下文 需每次携带 Agent 自动管理
使用成本 每次调用带上下文 按需调用
平台支持 通用 需 Agent 框架支持

MCP（Model Context Protocol）是 AI Agent 的标准协议，让 AI Agent 能够动态调用外部工具，实现真正的”实时查询”能力。

5.4.3 与 hiagent 的集成

hiagent 可以直接调用字段户口簿的后台 API，在 生成 Schema 时就完成字段解析，生成纯净的 Schema 和代码。

核心流程：

为什么在生成 Schema 时解析：

方案 Schema 内容 代码生成 优点
旧方案 带 mapping 选系统 一次 Schema 多系统复用
新方案 ⭐ 纯净字段 直接生成 Schema 简洁、无冗余逻辑

设计原则：

  • Schema 是需求的抽象，不是配置
  • 字段户口簿负责解析，生成时就已经确定
  • 代码是 Schema 的实现，无需任何映射逻辑

5.5 安全与合规设计

flowchart LR
    subgraph Security["安全机制"]

    subgraph Cache["分级缓存"]

    Cache --> L1

  • 服务身份认证：每个调用方（如 PRD2Code）需携带唯一的 serviceId 和 apiKey
  • 权限校验：实时向数据治理平台校验”该服务是否有权查询某字段”
  • 审计日志：所有查询记录（谁、查了什么、何时）落审计库，可追溯
  • 数据分级缓存：
    • 非敏感信息（字段类型、别名）→ L1 内存缓存
    • 通用信息 → L2 Redis 缓存
    • 敏感信息（是否涉敏、脱敏规则）→ 必须实时查询

  1. 字段户口簿从治理平台定时同步权威字段定义（字段 ID、类型、敏感等级等）
  2. 治理平台为户口簿开通受信 API，用于实时权限校验
  3. 户口簿维护系统别名（这是治理平台通常不做的），作为补充
  4. 户口簿对外提供 AI 友好接口，所有查询均附带服务身份，由治理平台做最终授权

7.2 核心差异

对比维度 数据治理平台 字段户口簿
核心定位 企业级元数据管理 AI 代码生成的语义底座
用户 数据治理人员、分析师 PRD2Code、AI IDE、开发者
数据来源 权威、全面 从治理平台同步+维护别名映射
接口风格 面向人（UI、报表） 面向机器（API、毫秒级响应）
更新频率 低频（按需变更） 高频（随时同步别名）
安全控制 强、合规 继承治理平台权限，不绕过

八、落地路径构思

第一阶段：MVP

  • 范围：仅一个业务域（如订单域），3-5 个核心字段
  • 功能：字段定义、别名映射、枚举值查询（HTTP 接口）
  • 数据来源：手动录入+访谈
  • 消费方：PRD2Code 内部试用
  • 成功标准：PRD2Code 生成订单页面时字段名不再出错

第二阶段：扩展

  • 范围：覆盖 2-3 个业务域（订单、客户、产品）
  • 功能：增加敏感字段查询、批量解析接口
  • 数据来源：对接公司已有的数据治理平台（如有）
  • 消费方：PRD2Code 正式使用，其他内部工具开始接入
  • 成功标准：至少 2 个业务线反馈联调效率提升 30%以上

第三阶段：企业级（如果可能的话）

  • 范围：覆盖全公司核心业务域
  • 功能：权限校验对接、审计日志、IDE 插件
  • 数据来源：与数据治理平台双向同步
  • 消费方：AI IDE（如 Cursor 插件）、所有内部系统
  • 成功标准：成为公司”AI 代码生成”的事实标准

九、来个总结吧

核心比喻

字段户口簿是「字典」，Schema 是「文章」。📖

  • 字典定义词条的标准写法
  • 文章引用字典中的词条
  • 查字典 → 写文章 → 出代码

价值总结

业务创新的最大阻力往往不是技术做不到，是系统之间语言不通。

字段户口簿就是把所有系统的字段别名收拢起来，建立一套标准。以后任何系统接入，只要查户口簿，自动翻译。

这不是技术工具，这是业务语义的共识层。

短期它不赚钱，但它让所有业务系统说同一种语言。没有它，公司越大，内耗越重，创新越慢。

现在投入一个 3 人团队，一年后就能跑通，两年后就是行业标准。

行业里都在做数据治理，但那是面向人的报表和仪表盘。下一波浪潮是面向机器的数据服务——AI 要能自动读懂公司数据。

字段户口簿就是提前把公司数据的”方言词典”建好，等 AI 时代真正到来时，我们的 AI 比别人聪明一个量级。

它现在不产生收益，但它是未来十年的技术债还清计划。
