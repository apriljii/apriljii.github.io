三、Vite 构建体系探索

3.1 Vite 尝试的背景

MS UI System 在发展过程中，为了提升开发体验和构建效率，曾经尝试引入 Vite 作为构建工具。Vite 以其快速的冷启动和热重载闻名，对现代化前端开发非常友好。

尝试目标

  1. 提升开发体验：利用 Vite 的快速热重载
  2. 简化配置：减少复杂的 webpack 配置
  3. 现代化构建：采用基于 ESModule 的构建方式
  4. 生态集成：利用 Vite 丰富的插件生态

3.3 遇到的技术挑战

Vue 2 兼容性问题

主要问题

  1. 热重载稳定性：Vue 2 项目中热重载有时会出现状态丢失
  2. 依赖预构建：某些 Node.js 内置模块处理不当
  3. 样式处理：复杂主题系统的样式热重载不够稳定
  4. 构建产物：UMD 格式输出在某些环境下有兼容性问题

3.4 回归 Webpack 的决策

经过一段时间的 Vite 尝试，我们发现虽然 Vite 在开发体验上有优势，但在生产构建的稳定性和对复杂项目的支持上，Webpack 仍然是更成熟的选择。

决策因素

  1. 项目成熟度：Webpack 生态更成熟，问题解决方案更多
  2. 构建稳定性：Webpack 对复杂构建场景的支持更稳定
  3. 团队熟悉度：团队对 Webpack 配置更熟悉
  4. 维护成本：Vite 的 Vue 2 支持相对较新，长期维护成本更高

6.1 当前构建策略

七、技术选型与决策反思

7.1 Vue 2 + Vite 的实际技术挑战

在 MS UI System 的实践中，我们发现 Vue 2 + Vite 的组合虽然在开发体验上有优势，但在生产环境和企业级应用场景中存在一些不可忽视的技术挑战：

1. 插件生态成熟度问题

2. 生产构建稳定性问题

3. 企业级功能缺失

7.2 从 Vite 回归 Webpack 的决策过程

基于上述技术挑战和企业级项目的需求，我们重新评估了技术选型，最终决定从混合模式逐步回归到 Webpack 主导的构建体系：

决策关键因素

  1. 生产环境稳定性优先：企业级项目对构建结果的稳定性和可预测性要求更高
  2. 生态系统完整性：Webpack 在 Vue 2 生态中的支持更加完善和成熟
  3. 团队维护成本：统一的构建工具链降低了学习和维护成本
  4. 现有投资保护：充分利用已有的 Webpack 配置和优化经验

平滑迁移策略

八、总结

虽然最初尝试了 Vite + Vue 2 的组合，但在实际应用中遇到了一些技术挑战，最终选择了更保守稳定的 Webpack 方案。

构建工具链演进历程：

  1. 初始阶段：纯 Webpack 构建，稳定可靠但开发体验一般
  2. 探索阶段：引入 Vite 提升开发体验，但遇到 Vue 2 兼容性问题
  3. 调整阶段：从混合模式逐步向 Webpack 回归
  4. 稳定阶段：建立 Webpack + 开发优化 的成熟方案

针对 MS UI System 的构建工具建议：

当前混合模式虽然提供了开发便利（当然主要原因还是 Vite 配置上的一些试错，迁移回 Webpack），但建议统一使用 Webpack，主要原因包括：

  1. 生产构建的稳定性和功能完整性更重要：Webpack 在企业级构建需求支持方面更成熟
  2. Vue 2 生态对 Webpack 支持更完善：经过多年验证，问题更少，文档更全
  3. 避免维护两套构建系统的复杂度：减少团队学习成本和维护负担

技术决策的关键考虑：

  • Vue 2 组件库的特殊性：与 Vue 3 项目不同，Vue 2 对构建工具的兼容性要求更严格
  • 企业级项目的稳定性要求：生产环境的稳定性和可维护性优先于开发体验的提升
  • 团队技术栈的一致性：避免引入过多新技术栈导致的学习和维护成本

未来优化方向：

  1. 构建配置优化：持续优化 Webpack 配置，提升构建效率
  2. 多格式支持完善：完善 ESM、UMD 等多种构建格式
  3. 构建脚本改进：优化构建脚本和错误处理
  4. 构建文档完善：建立清晰的构建配置说明和使用指南

在企业级 Vue 2 项目中，技术稳定性和维护便利性往往比开发体验更重要。
